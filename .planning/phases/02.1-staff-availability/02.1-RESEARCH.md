# Phase 2.1: Staff Availability - Research

**Researched:** 2026-02-25
**Domain:** Next.js App Router staff authentication, WorkerSchedule CRUD, server actions
**Confidence:** HIGH

## Summary

Phase 2.1 is a staff self-service page allowing each worker to view and edit their own recurring weekly schedule (dayOfWeek-based WorkerSchedule records). The codebase already has everything needed: the WorkerSchedule model exists and is fully wired into the booking system's slot logic. No schema changes are required.

The primary architectural decision is authentication: the project currently has no staff auth. The Worker model has no login credentials, so a new auth mechanism is needed. The simplest approach that fits the existing patterns is a PIN-based or shared-secret JWT similar to how admin auth works — storing a `workerId` in a signed cookie — without adding a password field to the Worker model. An alternative is to extend the existing admin auth to expose a staff-scoped view (admins access the page on behalf of workers). The second option is significantly lower effort and avoids introducing a new auth surface.

The UI work is straightforward: a 7-row weekly grid (one per day of week) with start/end time selectors and an on/off toggle per day. The project uses custom UI components (Input, Button, Card), native `<select>` elements for time pickers, `useActionState` for form handling, and `next-intl` for translations. All these are already established.

**Primary recommendation:** Build the schedule editor under the existing admin auth as a sub-page within `/admin/workers/[id]/schedule` — this avoids a new auth system entirely and the admin acts as the staff manager. Staff log in as admin (single shared credential) and navigate to their own schedule. If true per-worker isolation is required (success criterion 5), implement a cookie-based `workerId` JWT — same pattern as admin — with a simple PIN or shared link mechanism.

### Pattern 1: WorkerSchedule Upsert by dayOfWeek

**What:** Upsert all 7 WorkerSchedule records for a given worker in a single server action.
**When to use:** When the staff member saves their weekly schedule.

The WorkerSchedule schema allows one record per `(workerId, dayOfWeek)` for recurring availability. The seed data confirms this: `id: schedule-${worker.id}-${dayOfWeek}`. However, the schema does NOT have a `@@unique([workerId, dayOfWeek])` constraint — upsert must use `id` as the lookup key, OR use `findFirst` + `update`/`create` pattern.

**Upsert approach (using id):**
```typescript
// Source: prisma/seed.ts — established project pattern
await prisma.workerSchedule.upsert({
  where: { id: `schedule-${workerId}-${dayOfWeek}` },
  update: { startTime, endTime, isAvailable },
  create: {
    id: `schedule-${workerId}-${dayOfWeek}`,
    workerId,
    dayOfWeek,
    startTime,
    endTime,
    isAvailable,
    specificDate: null,
  },
})
```

**Alternative (no assumed id format):**
```typescript
const existing = await prisma.workerSchedule.findFirst({
  where: { workerId, dayOfWeek, specificDate: null },
})
if (existing) {
  await prisma.workerSchedule.update({
    where: { id: existing.id },
    data: { startTime, endTime, isAvailable },
  })
} else {
  await prisma.workerSchedule.create({
    data: { workerId, dayOfWeek, startTime, endTime, isAvailable, specificDate: null },
  })
}
```

The findFirst approach is safer because it does not rely on the seed's id convention. Recommend this for production robustness.

### Pattern 5: Week Day Rendering

The WorkerSchedule model uses `dayOfWeek: Int` where 0=Sunday, 6=Saturday. The UI grid renders days 0–6 in Japanese order (Sun–Sat or Mon–Sun). The existing availability route uses `new Date(dateStr).getUTCDay()` for day-of-week lookup — use the same convention (0=Sunday).

### Anti-Patterns to Avoid

- **Batch vs. per-field saves:** Do not make one server action per day. Submit all 7 days in a single form/action to avoid partial-save states.
- **Using `specificDate` records:** This phase is dayOfWeek only. Do not create `specificDate` rows here — those are for one-off overrides.
- **Optimistic updates without revalidation:** Always call `revalidatePath` after mutations. The booking system reads directly from DB; stale cache would serve wrong availability.
- **Missing `isAvailable: true` filter:** The availability route (`/api/availability`) already filters `isAvailable: true`. If a day is toggled off, set `isAvailable: false` rather than deleting the record — this preserves history and is consistent with how the schema is designed.
- **No uniqueness constraint on `(workerId, dayOfWeek)`:** The schema has no unique index on this pair (only on `(workerId, specificDate)`). The findFirst+update/create pattern is required; a raw upsert without a unique constraint will fail.

## Common Pitfalls

### Pitfall 1: No Unique Constraint on (workerId, dayOfWeek)

**What goes wrong:** A developer uses `prisma.workerSchedule.upsert({ where: { workerId_dayOfWeek: ... } })` expecting a compound unique — this will fail because no such unique index exists in the schema.

**Why it happens:** The schema has `@@index([workerId, dayOfWeek, startTime, endTime])` (an index, not a unique constraint). The developer assumes index = unique.

**How to avoid:** Use `findFirst({ where: { workerId, dayOfWeek, specificDate: null } })` then branch on update/create.

**Warning signs:** Prisma TypeScript errors on `where` clause for upsert.

### Pitfall 2: Mixing specificDate and dayOfWeek Records

**What goes wrong:** The schedule editor creates `specificDate: null` records but the `findMany` query fetches all schedules including `specificDate: someDate` ones. The UI shows stale one-off overrides as if they are the recurring schedule.

**Why it happens:** The schedule page doesn't filter `specificDate: null` in its query.

**How to avoid:** Always add `specificDate: null` to the `where` clause when fetching recurring schedules:
```typescript
prisma.workerSchedule.findMany({
  where: { workerId, specificDate: null, dayOfWeek: { not: null } }
})
```

**Warning signs:** A worker's displayed schedule doesn't match what bookings actually use.

### Pitfall 3: endTime <= startTime Validation Gap

**What goes wrong:** A worker sets startTime=18:00 and endTime=09:00 (valid strings, invalid range). The availability service generates zero slots (correct), but the UI shows no error. The worker thinks they are available but no slots appear.

**Why it happens:** No Zod `refine` validation that endTime > startTime.

**How to avoid:** Add to Zod schema:
```typescript
.refine(data => data.endTime > data.startTime, {
  message: 'End time must be after start time',
  path: ['endTime'],
})
```

**Warning signs:** Worker saves schedule successfully but sees no slots on the booking calendar.

### Pitfall 4: Stale Booking System After Schedule Change

**What goes wrong:** Admin changes worker's schedule, but the `/api/availability` endpoint serves stale data because Next.js cached the route.

**Why it happens:** The availability API route uses `prisma.workerSchedule.findMany` — if Next.js caches the response, schedule changes are invisible until cache expires.

**How to avoid:** Ensure `revalidatePath` or `revalidateTag` is called in the server action. The API route at `/api/availability` is a dynamic API route (no static generation) so this should not be an issue with default config, but worth verifying.

**Warning signs:** Booking calendar shows old slots after a schedule change.

### Pitfall 5: Worker ID from URL vs. Session

**What goes wrong:** If this is implemented as admin-only, the `workerId` comes from the URL param `[id]`. If staff auth is added later, the `workerId` must come from the session, not URL, to prevent one staff member editing another's schedule.

**Why it happens:** URL-based IDs are easy to tamper with.

**How to avoid:** If the page is admin-only, validate that the URL `workerId` corresponds to an existing active worker. If staff auth is added, derive `workerId` from the JWT payload — never trust URL params alone for staff-scoped access.

### Upsert Schedule Day (Server Action)

```typescript
// Source: app/actions/workers.ts pattern + seed.ts upsert pattern
'use server'
import { revalidatePath } from 'next/cache'
import { prisma } from '@/lib/db/client'
import { getAdminSession } from '@/lib/auth/admin'

export async function upsertWorkerSchedule(
  workerId: string,
  schedules: Array<{
    dayOfWeek: number
    startTime: string
    endTime: string
    isAvailable: boolean
  }>
) {
  const isAdmin = await getAdminSession()
  if (!isAdmin) throw new Error('Unauthorized')

  for (const schedule of schedules) {
    const existing = await prisma.workerSchedule.findFirst({
      where: { workerId, dayOfWeek: schedule.dayOfWeek, specificDate: null },
    })

    if (existing) {
      await prisma.workerSchedule.update({
        where: { id: existing.id },
        data: {
          startTime: schedule.startTime,
          endTime: schedule.endTime,
          isAvailable: schedule.isAvailable,
        },
      })
    } else {
      await prisma.workerSchedule.create({
        data: {
          workerId,
          dayOfWeek: schedule.dayOfWeek,
          startTime: schedule.startTime,
          endTime: schedule.endTime,
          isAvailable: schedule.isAvailable,
          specificDate: null,
        },
      })
    }
  }

  revalidatePath(`/admin/workers`)
  return { success: true }
}
```

### Zod Validation Schema

```typescript
// Source: src/lib/validations/worker.ts pattern
import { z } from 'zod'

const dayScheduleSchema = z.object({
  dayOfWeek: z.number().int().min(0).max(6),
  startTime: z.string().regex(/^\d{2}:\d{2}$/),
  endTime: z.string().regex(/^\d{2}:\d{2}$/),
  isAvailable: z.boolean(),
}).refine(
  data => !data.isAvailable || data.endTime > data.startTime,
  { message: 'End time must be after start time', path: ['endTime'] }
)

export const workerScheduleSchema = z.array(dayScheduleSchema).length(7)
```

## Open Questions

1. **True per-worker auth isolation required?**
   - What we know: Success criterion 5 says "page accessible with staff authentication" — implying workers log in individually, not via shared admin credentials.
   - What's unclear: The Worker model has no credentials (no email, no password). A new auth mechanism is needed if per-worker isolation is required.
   - Recommendation: Clarify with product. Two options:
     - **Option A (low effort):** Admin-only page at `/admin/workers/[id]/schedule`. Admin manages all worker schedules. No staff auth needed.
     - **Option B (medium effort):** Add a `pinCode` or `accessCode` field to Worker. Create `/staff/login` → JWT cookie with `{ workerId, role: 'staff' }` → protected `/staff/schedule` page. Mirror admin auth pattern from `src/lib/auth/admin.ts`.
   - This decision determines whether a schema migration is needed (Option B) or not (Option A).

2. **Single-page bulk save vs. per-day auto-save?**
   - What we know: The admin pattern uses explicit save buttons. The user booking flow has no auto-save.
   - What's unclear: Product UX preference.
   - Recommendation: Use a single "Save all" button — simpler, avoids race conditions with 7 parallel saves.

3. **Sunday handling in week display?**
   - What we know: `dayOfWeek: 0` = Sunday, matching JavaScript's `Date.getDay()`. Seed creates schedules for Mon-Sat (1–6) only.
   - What's unclear: Does the business operate on Sundays? Should the UI show all 7 days or only Mon-Sat?
   - Recommendation: Show all 7 days with toggle. Workers can mark Sunday as unavailable. Follows the schema's full 0-6 range.
