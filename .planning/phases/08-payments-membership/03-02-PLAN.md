  <action>
    1. Create `src/lib/services/payment.service.ts`:

       **`handleStripeEvent(event: Stripe.Event)`**: Main event dispatcher
       - Idempotency check: `prisma.stripeEvent.findUnique({ where: { stripeEventId: event.id } })` — if found AND `processedAt` is not null, return early (already processed)
       - If not found, create StripeEvent record with `processedAt: null` (marks as "in progress")
       - Process the event (switch on event.type)
       - On SUCCESS: update StripeEvent to set `processedAt: new Date()` — this marks it as done
       - On FAILURE: delete the StripeEvent record (allows Stripe to retry)
       - This pattern ensures: duplicates are skipped, but failed processing can be retried

       Switch on `event.type`:
         - `checkout.session.completed`: call `handleCheckoutComplete(session)`
         - Default: log unhandled event type and return (still mark as processed)

       **`handleCheckoutComplete(session: Stripe.Checkout.Session)`**: Inside a `prisma.$transaction`:
       - Extract `internalCustomerId` from `session.metadata`
       - Extract `planId` from `session.metadata`
       - Extract `stripeCustomerId` from `session.customer` (cast to string)
       - Update Customer: set `stripeCustomerId` (only if not already set)
       - Find MembershipPlan by planId
       - Create Membership record: customerId, planId, sessionsRemaining: plan.sessionsIncluded, status: 'active'
       - Create Payment record: customerId, stripeSessionId: session.id, amount: session.amount_total, currency: 'jpy', status: 'succeeded', description: `${plan.name} (${plan.sessionsIncluded}回)`

       **`createCheckoutSession(customerId, planId, origin, locale)`**: For use by checkout API
       - Look up MembershipPlan by planId to get stripePriceId
       - Look up Customer by customerId to get email
       - Call `getStripe().checkout.sessions.create()` with:
         - `mode: 'payment'` (one-time, NOT subscription)
         - `line_items: [{ price: stripePriceId, quantity: 1 }]`
         - `customer_email: customer.email`
         - `metadata: { internalCustomerId: customerId, planId }` (top-level metadata is fine for payment mode)
         - `success_url: \`${origin}/${locale}/membership/success?session_id={CHECKOUT_SESSION_ID}\``
         - `cancel_url: \`${origin}/${locale}/membership/cancel\``
       - Return session.url
       - Note: Apple Pay works automatically with Stripe hosted checkout

       **`getPaymentsByCustomer(customerId, page, limit)`**: Paginated payment history
       - Query Payment where customerId, orderBy createdAt desc, skip/take for pagination
       - Return { payments, total }

    2. Create `src/lib/services/membership.service.ts`:

       **`getMembershipPlans(activeOnly?: boolean)`**: List plans
       - Query MembershipPlan, optionally filter isActive: true, orderBy priceJpy asc
       - Return plans array

       **`getMembershipPlanById(planId: string)`**: Single plan
       - Query by id, return plan or null

       **`getActiveMembership(customerId: string)`**: Get customer's active membership (has sessions left)
       - Query Membership where customerId AND status: 'active' AND sessionsRemaining > 0, include plan, orderBy purchasedAt desc, take 1
       - Return membership with plan details or null

       **`getMembershipByCustomer(customerId: string)`**: All memberships for a customer
       - Query Membership where customerId, orderBy purchasedAt desc, include plan
       - Return array

       **`createMembershipPlan(data)`**: Admin create plan
       - Create MembershipPlan record from validated data
       - Return created plan

       **`updateMembershipPlan(id, data)`**: Admin update plan
       - Update MembershipPlan by id
       - Return updated plan

       **`deactivateMembershipPlan(id)`**: Admin soft-delete
       - Update isActive to false
       - Return updated plan

       **`decrementSession(membershipId: string)`**: Decrement session on QR scan
       - Use `prisma.membership.update` with `sessionsRemaining: { decrement: 1 }`
       - After decrement, check if sessionsRemaining is now 0 → set status to 'exhausted'
       - Return updated membership
       - Important: This should be atomic — use a transaction or Prisma's atomic decrement

    Follow existing service patterns from `booking.service.ts` and `customer.service.ts`.

  <action>
    Create `app/api/stripe/webhook/route.ts`:

    1. Export `POST` handler:
       - CRITICAL: Use `await request.text()` to get raw body — NOT `request.json()`. Required for Stripe signature verification.
       - Get `stripe-signature` header via `request.headers.get('stripe-signature')`
       - If no signature, return 400
       - Call `getStripe().webhooks.constructEvent(body, signature, process.env.STRIPE_WEBHOOK_SECRET!)` in try/catch
       - On verification failure, log error and return 400 with "Invalid signature"
       - Call `handleStripeEvent(event)` from payment.service.ts in try/catch
       - On processing error:
         - If error is retryable (DB connection timeout, network error): return 500 so Stripe retries
         - If error is permanent (data validation, missing references): log and return 200 to stop retries
         - Distinguish by error type: Prisma connection errors → 500, all others → 200
       - On success: Return `NextResponse.json({ received: true }, { status: 200 })`

    2. This route must NOT have any authentication middleware — Stripe needs to call it directly. Signature verification IS the auth.

    3. Export `const dynamic = 'force-dynamic'` to prevent caching.
  </action>

    - Retryable errors return 5xx, permanent errors return 200
  </verify>
